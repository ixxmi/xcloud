# xcloud 中文详细文档

`xcloud` 是一个中心化文件同步 MVP。它的目标是让多台安装了客户端程序的设备，通过一个服务端文件中心同步指定目录内的文件。

当前版本采用“服务端协调 + 客户端扫描同步 + 内容分块存储”的设计：

```text
Client A ─┐
Client B ─┼── xcloud Server ── metadata.json
Client C ─┘          │
                     └── chunks/<hash>
```

服务端负责认证、元数据、版本协调、冲突判断、删除墓碑和分块存储。客户端负责扫描本地目录、计算 hash、上传缺失块、提交版本、拉取远端变更并安全落盘。

现在服务端还包含一个管理后台。后台用于登录、注册、创建账号、修改密码、管理客户端上报的本地目录、创建 Space、重置同步 token。客户端启动后先把本地目录信息上报给网关，网关在后台把目录选择到某个 Space 后，同一账号、同一 Space 下的目录才会互相同步。

## 当前能力

- 一个二进制同时支持服务端和客户端模式。
- 管理后台支持登录、注册、账号、密码、客户端目录、Space 和同步 token 管理。
- 每个账号拥有独立同步 token。
- 每个账号可以创建多个 Space，用于把不同项目或目录组隔离开。
- 客户端可以上报任意指定目录，但必须由网关选择后才会参与同步。
- 同步严格限制在同一账号、同一 Space 内。
- 服务端保存文件版本元数据和内容分块。
- 文件和分块均使用 SHA-256 校验。
- 上传前客户端先询问服务端缺失哪些块，只上传缺失块。
- 服务端会按已上传分块重新计算文件大小和 hash，不信任客户端直接上报的文件 hash。
- 下载时客户端校验每个分块 hash，写入完成后再校验整文件 hash。
- 下载落盘使用临时文件 + 原子 rename，避免写一半的损坏文件覆盖原文件。
- 支持删除 tombstone。删除不是直接从服务端元数据消失，而是作为一个版本事件传播。
- 支持本地冲突保留。遇到两端同时修改时，不直接覆盖本地未同步内容。
- 默认跳过符号链接、特殊文件和客户端状态目录 `.xcloud`。
- 支持 Bearer token 鉴权。

## 构建

在项目根目录执行：

```sh
go build -o xcloud ./cmd/xcloud
```

如果本机 Go 构建缓存目录受限，可以临时指定缓存目录：

```sh
GOCACHE=/tmp/go-build-cache go build -o xcloud ./cmd/xcloud
```

## 启动服务端

```sh
./xcloud server -addr :8080 -data ./server-data
```

参数说明：

```text
-addr   HTTP 监听地址，默认 :8080
-data   服务端数据目录，默认 ./xcloud-data
```

服务端数据目录示例：

```text
server-data/
├── metadata.json
└── chunks/
    └── ab/
        └── cd/
            └── <sha256>
```

`metadata.json` 保存文件树、版本、事件和幂等操作结果。`chunks` 保存按内容 hash 命名的文件块。

## 管理后台

服务端启动后访问：

```text
http://127.0.0.1:8080/admin
```

后台功能：

- 用户名或邮箱登录。
- 普通账号自助注册。
- 当前账号修改登录密码。
- 创建普通账号或管理员账号。
- 为账号重置同步 token。
- 查看客户端上报的本地目录。
- 把客户端目录选择到某个 Space。
- 为账号创建 Space。
- 停用或启用账号。
- 停用客户端目录或 Space。
- 查看每个 Space 的目录数量、文件数量和删除记录数量。

同步 token 只在创建账号或重置 token 时显示一次。客户端不使用登录密码同步，而是使用账号的同步 token。

普通注册账号会自动创建 `default` Space，并在注册成功后自动登录。管理员创建账号时也会为该账号创建 `default` Space。

生产环境必须使用安全的初始化方式处理管理员账号，并且必须启用 TLS。当前 MVP 仍会为首次启动创建默认管理员，部署时应立即修改密码或改造初始化流程。

## 账号、客户端目录和 Space

权限隔离规则：

```text
账号 A + space default  <=>  账号 A + space default
账号 A + space docs     <=>  账号 A + space docs
账号 B + space default  <=>  账号 B + space default
```

不同账号之间永远不会同步。即使两个账号使用同名 `default` 空间，也属于不同的数据域。

同一账号下不同 Space 也不会互相同步。例如：

```text
账号 alice / space docs
账号 alice / space photos
```

这两个 Space 的数据互相隔离。

客户端目录的生命周期：

```text
客户端启动
  -> 先上报根级目录，例如当前工作目录和用户主目录
  -> 网关记录为 pending
  -> 后台点击“展开下一级”或“选择同步”
  -> 网关要求客户端上报该目录的直接子目录
  -> 客户端下一轮只上报被请求目录的下一级，并标记已上报
  -> 后台选择某个目录到 Space
  -> 客户端下一轮得到 selected
  -> 开始上传、下载和接收同账号同 Space 的事件
```

`-space` 只是客户端上报时给网关的建议分组。真正生效的 Space 是管理后台选择的 Space。网关未选择前，客户端不会上传本地文件，也不会下载远端文件。已经上报过的目录会记录在客户端发现状态里，后续不会重复上报。

## 启动客户端

首次启动客户端不需要手工携带 token。客户端会启动一个只监听本机的管理页面：

```sh
./xcloud client -server http://127.0.0.1:8080
```

打开 `http://127.0.0.1:18080`，输入云端账号和密码登录，然后点击 `开启此账号同步`。客户端会向云端换取本机专用同步凭证，并保存到：

```text
~/.xcloud/client-config.json
```

开启同步是账号级云端开关：任意一台已经登录该账号的客户端点击开启后，其他已经登录同一账号的客户端会自动识别到账号已开启同步，并立即开始上报目录和执行同步。云端 `/admin` 的 `目录与 Space` 页面选择目录到 Space 后，同账号同 Space 的目录才会真正上传、下载和互相同步。

如果不传 `-root`，客户端默认使用进程启动目录下的 `xcloud` 目录作为同步放置目录，并自动创建该目录。管理后台可以为任意已上报客户端目录修改本地同步放置目录；客户端下一轮状态刷新后会使用新目录落盘和扫描。

如果启动时加了 `-root`，本地管理页面仍然可用，但同步范围只限这个目录：

```sh
./xcloud client -root /tmp/xcloud-a -server http://127.0.0.1:8080
```

旧的 token 启动方式仍然保留，适合脚本和系统服务部署。

准备两个目录模拟两台设备：

```sh
mkdir -p /tmp/xcloud-a /tmp/xcloud-b
```

启动客户端 A：

```sh
XCLOUD_TOKEN=secret ./xcloud client \
  -root /tmp/xcloud-a \
  -server http://127.0.0.1:8080 \
  -space default \
  -device laptop-a
```

启动客户端 B：

```sh
XCLOUD_TOKEN=secret ./xcloud client \
  -root /tmp/xcloud-b \
  -server http://127.0.0.1:8080 \
  -space default \
  -device laptop-b
```

`-root` 现在不是必填。客户端有两种模式：

- 不传 `-root`：客户端使用 `<进程启动目录>/xcloud` 作为默认同步目录并上报；后台展开某个目录后，客户端才会上报它的下一级目录。
- 传 `-root`：客户端进入单目录模式，只上报并同步这个指定目录，兼容明确指定目录的部署方式。

客户端默认使用文件系统实时监听触发同步，并保留周期性扫描作为兜底。管理后台的 `同步设置` 菜单可以配置是否启用实时监听、变更防抖毫秒数和兜底扫描间隔秒数。但第一轮会先上报本地目录到服务端。如果后台还没有把目录选择到某个 Space，客户端会停在等待状态，不上传、不下载。

第一次接入目录的步骤：

1. 启动客户端 A 和客户端 B；可以不传 `-root` 让它们上报候选目录，也可以传 `-root` 指定单个目录。
2. 打开管理后台 `/admin`。
3. 进入 `目录与 Space` 菜单。
4. 先看到根级目录；需要更细粒度目录时，点击 `展开下一级`。
5. 客户端下一轮上报子目录后，在列表中选择要同步的目录。
6. 把 A、B 的目标目录选择到同一个 Space，例如 `default`。
7. 客户端开始在这两个目录之间同步文件；后续文件变化会实时触发同步。

管理后台的 `同步记录` 菜单会显示上传、下载、删除、冲突、跳过和失败记录。记录按文件操作保存，包含账号、Space、设备、根目录、文件路径、状态、耗时和错误信息。

常用参数：

```text
-root           可选；指定后只上报并同步该目录，不指定则分层发现目录，由后台逐级展开
-server         服务端地址，默认 http://127.0.0.1:8080
-token          可选；旧的 Bearer token 启动方式，也可以通过 XCLOUD_TOKEN 设置；不传则启动本地管理页面
-client-addr    本地客户端管理页面地址，默认 127.0.0.1:18080
-client-config  本地客户端配置路径，默认 ~/.xcloud/client-config.json
-space          建议 Space ID，也可以通过 XCLOUD_SPACE 设置，默认 default；最终以网关选择为准
-device         当前设备 ID，也可以通过 XCLOUD_DEVICE_ID 设置
-state          单目录模式下的客户端状态文件路径，默认 <root>/.xcloud/state.json
-interval       扫描同步间隔，默认 10s
-chunk-size     分块大小，默认 4194304，即 4 MiB
-once           只执行一轮同步后退出
-delete-remote  将本地删除传播到服务端
```

## 一次性同步

调试时可以使用 `-once`，只运行一轮同步：

```sh
XCLOUD_TOKEN=secret ./xcloud client \
  -root /tmp/xcloud-a \
  -server http://127.0.0.1:8080 \
  -space default \
  -device laptop-a \
  -once
```

## 删除同步

为了避免误删扩散，客户端默认不把本地删除传播到服务端。也就是说，如果你删除了本地文件，默认情况下客户端不会主动告诉服务端“这个文件应该删除”。

如果希望本地删除同步到其他设备，启动客户端时加：

```sh
-delete-remote
```

删除同步流程：

```text
本地文件消失
客户端发现 state 中有记录但扫描不到文件
客户端向服务端提交 delete 请求
服务端生成 deleted 版本
其他客户端拉取事件后删除本地文件
```

删除版本仍然保存在服务端元数据中，便于后续实现历史版本、回滚和误删恢复。

## 同步流程

客户端每轮同步先确认本地目录是否已经被网关选择：

```text
1. reportFolder 上报本地目录并读取网关选择结果
2. pullRemote   拉取服务端事件并应用到本地
3. pushLocal    扫描本地目录并上传本地变化
4. pullRemote   再拉一次，处理上传过程中出现的新远端变化
```

如果第 1 步返回未选择，后续步骤不会执行。

### 本地扫描

单目录模式下，客户端遍历 `-root` 指定目录。目录发现模式下，客户端只会遍历已经被网关选择的目录：

- 跳过 `.xcloud` 状态目录。
- 跳过符号链接。
- 跳过目录、设备文件、socket、pipe 等特殊文件。
- 只处理普通文件。
- 对每个普通文件计算整文件 SHA-256。
- 按 `-chunk-size` 分块并计算每个块的 SHA-256。

### 上传流程

```text
客户端扫描文件
客户端计算 manifest
客户端调用 /v1/chunks/check
服务端返回缺失的 chunk hash
客户端上传缺失 chunk
客户端调用 /v1/commit 提交版本
服务端校验 manifest
服务端生成新版本和事件
```

manifest 结构概念如下：

```json
{
  "path": "docs/a.txt",
  "base_version": "旧版本 ID",
  "size": 123,
  "hash": "整文件 sha256",
  "chunks": [
    {
      "index": 0,
      "hash": "chunk sha256",
      "size": 123
    }
  ],
  "mod_time_unix": 1710000000
}
```

服务端提交版本时会重新读取对应 chunk，重新计算整文件 size 和 hash。只有服务端计算结果与 manifest 一致时，版本才会提交成功。

### 下载流程

```text
客户端拉取服务端事件
发现远端有新 file 版本
检查本地是否存在未同步修改
如有未同步修改，先改名保留为冲突副本
下载所有 chunk 到临时文件
校验 chunk hash
校验整文件 hash
原子 rename 覆盖目标文件
更新本地 state
```

## 版本模型

每个文件有一个稳定的 `file_id`，每次修改生成一个新的 `version_id`。

客户端提交修改时会带上本地已知的 `base_version`。服务端用它判断这次提交是否基于当前最新版本。

```text
base_version == 服务端当前版本
=> 正常提交

base_version != 服务端当前版本
=> 说明服务端已经有更新版本，当前提交会变成冲突副本
```

这种方式避免“最后写入者直接覆盖前一个写入者”的数据丢失问题。

## 冲突处理

当前版本不做文本三方合并，而是采用更保守的方式：保留两份文件。

### 服务端提交冲突

如果客户端提交文件时，服务端发现它基于的版本不是当前版本，会生成一个新的冲突路径：

```text
report.docx
report (conflict from laptop-b at 20260516-103000).docx
```

这样服务端不会覆盖当前主路径文件。

### 客户端拉取冲突

如果客户端准备应用远端版本，但发现本地文件已经被用户改过且尚未同步，会先把本地文件改名：

```text
report.docx
report (local conflict from laptop-a at 20260516-103000).docx
```

然后再把远端版本下载到原路径。这样本地未同步内容不会丢失。

## 客户端状态文件

客户端默认在同步目录下保存：

```text
<root>/.xcloud/state.json
```

它记录：

- 本地路径。
- file_id。
- version_id。
- 文件状态，file 或 deleted。
- 文件大小。
- 文件 hash。
- chunk 列表。
- 上次处理到的服务端事件序号。

客户端依赖这个状态判断本地文件是否变化、删除是否应该传播、远端事件是否已经处理过。

目录发现模式还会在用户主目录下保存：

```text
~/.xcloud/discovery-state.json
```

它记录已上报过的目录，避免客户端每一轮重复上报相同目录。单目录同步状态仍然保存在对应同步目录的 `.xcloud` 下。

不要手动编辑这个文件，除非你明确知道会造成什么结果。

## 服务端 HTTP API

当前服务端 API：

```text
GET  /healthz
POST /v1/folders/report
GET  /v1/files
GET  /v1/events?after=<seq>
POST /v1/chunks/check
PUT  /v1/chunks/<hash>
GET  /v1/chunks/<hash>
POST /v1/commit
POST /v1/delete
```

除 `/healthz` 外，其他接口如果配置了 token，都需要请求头：

```text
Authorization: Bearer <token>
X-XCloud-Space: <space_id>
X-XCloud-Device: <device_id>
X-XCloud-Root: <absolute_root_path>
```

`POST /v1/folders/report` 只要求 `Authorization`，`X-XCloud-Space` 可作为建议 Space，其他同步接口必须带 `X-XCloud-Device` 和 `X-XCloud-Root`。服务端会校验这个设备目录是否已经被选择到请求中的 Space。

`X-XCloud-Space` 为空时默认使用 `default`。

### POST /v1/folders/report

客户端上报本地目录：

```json
{
  "device_id": "laptop-a",
  "hostname": "my-host",
  "root_path": "/tmp/xcloud-a",
  "suggested_space_id": "default"
}
```

服务端返回目录状态。如果 `selected` 为 `false`，客户端应该等待后台选择；如果 `selected` 为 `true`，返回的 `space.id` 就是本轮同步应该使用的实际 Space。

### GET /v1/files

返回服务端当前文件列表，包括已删除条目的 tombstone 状态。

### GET /v1/events

返回某个事件序号之后的所有事件。客户端通过它增量拉取远端变化。

### POST /v1/chunks/check

客户端提交一组 chunk hash，服务端返回缺失的 hash。客户端只上传缺失块。

### PUT /v1/chunks/<hash>

上传一个 chunk。服务端会计算上传内容的 SHA-256，必须与 URL 中的 hash 一致。

### GET /v1/chunks/<hash>

下载一个 chunk。客户端下载后还会再次校验 hash。

### POST /v1/commit

提交一个文件版本。服务端会验证：

- operation_id 是否存在。
- device_id 是否存在。
- 路径是否是安全相对路径。
- 文件 hash 是否合法。
- 每个 chunk 是否存在。
- chunk index 是否连续。
- chunk size 是否匹配。
- 由 chunk 重新计算出的整文件 hash 是否匹配。
- base_version 是否与当前版本一致。

### POST /v1/delete

提交一个删除 tombstone。服务端会验证 base_version，避免删除掉已经被其他客户端更新过的文件。

## 安全设计

当前 MVP 已经实现了几项基础安全措施：

- Bearer token 鉴权。
- 同步 token 绑定账号。
- 文件、事件、版本和操作幂等结果按账号和 Space 隔离。
- 非上报接口必须携带设备 ID 和本地根目录，且该目录必须已经被网关选择到当前 Space。
- chunk 下载需要账号拥有该 chunk 的授权。
- 服务端路径校验，拒绝绝对路径、空路径、`.`、`..` 和路径穿越。
- 服务端校验 chunk hash。
- 服务端重新计算整文件 hash。
- 客户端下载后校验 chunk 和整文件 hash。
- 客户端跳过符号链接，避免把同步目录外的文件意外带入。
- 客户端写文件使用临时文件和原子 rename。

生产环境仍然必须补齐：

- TLS 1.3，避免 token 和文件内容明文传输。
- 更好的设备注册和吊销机制，避免所有设备共用一个 token。
- 每设备独立凭证或 mTLS 客户端证书。
- 服务端 API 限流和审计日志。
- Space 级更细粒度权限控制。
- 端到端加密，服务端只保存密文。
- 文件名和路径元数据加密，适用于高安全场景。

## 可靠性设计

当前版本已经具备以下可靠性基础：

- 客户端 state 记录已同步版本。
- 服务端 `operation_id` 支持幂等提交，避免客户端重试造成重复版本。
- 文件内容先上传 chunk，再提交 manifest。
- 只有 commit 成功后，其他客户端才会看到新事件。
- 下载文件先写临时文件，校验成功后原子替换。
- 删除作为版本事件保存，不直接从元数据抹掉。
- 冲突时保留副本，不静默覆盖。

后续建议继续增强：

- 客户端本地任务队列，明确记录 pending upload/download/delete。
- 服务端元数据迁移到 PostgreSQL，用事务保证提交一致性。
- chunk 存储迁移到 MinIO/S3，并增加生命周期管理。
- 增加历史版本保留策略。
- 增加快照和目录级回滚。
- 增加断点续传和并发上传下载。

## 当前限制

这版代码是 MVP，不是最终生产版。

已知限制：

- 客户端使用周期性扫描，还没有接入 fsnotify 实时监听。
- 服务端元数据使用单个 JSON 文件，不适合大规模并发和海量文件。
- Bearer token 是共享密钥，不支持设备级吊销。
- 管理后台 session 保存在服务端内存中，服务重启后需要重新登录。
- 当前没有找回密码和邮箱验证能力。
- 没有 TLS，需要通过反向代理或后续代码补齐。
- 没有端到端加密。
- 没有保留策略和垃圾回收，旧 chunk 暂不会自动清理。
- 没有文本三方合并。
- 没有带宽限速、并发控制和系统服务安装脚本。

## 推荐生产化路线

建议按下面顺序演进，而不是一次性堆功能：

1. 引入配置文件，支持多个同步目录。
2. 增加登录审计、后台 CSRF 防护、邮箱验证和找回密码。
3. 增加 fsnotify，实时监听变化，扫描作为兜底。
4. 增加 TLS/mTLS 和设备注册、吊销。
5. 把 metadata.json 迁移到 PostgreSQL。
6. 把本地 chunk 目录抽象成存储接口，支持 MinIO/S3。
7. 增加客户端任务队列和失败重试状态。
8. 增加历史版本列表和恢复命令。
9. 增加端到端加密。
10. 增加系统服务安装方式，例如 launchd、systemd、Windows Service。

## 本地验证流程

可以用下面流程验证两端同步。

启动服务端：

```sh
./xcloud server -addr :8080 -data /tmp/xcloud-server
```

打开 `/admin` 注册或登录账号，重置同步 token，下面命令里的 `secret` 替换为该 token。

创建两个客户端目录：

```sh
mkdir -p /tmp/xcloud-a /tmp/xcloud-b
```

在 A 写入文件：

```sh
echo "hello from a" > /tmp/xcloud-a/hello.txt
```

A 上传：

```sh
XCLOUD_TOKEN=secret ./xcloud client \
  -root /tmp/xcloud-a \
  -server http://127.0.0.1:8080 \
  -space default \
  -device a \
  -once
```

第一次执行时客户端只会上报目录并等待选择。打开后台 `目录与 Space`，把 A 目录选择到 `default`。

B 下载：

```sh
XCLOUD_TOKEN=secret ./xcloud client \
  -root /tmp/xcloud-b \
  -server http://127.0.0.1:8080 \
  -space default \
  -device b \
  -once
```

B 第一次执行同样只会上报目录。把 B 目录也选择到 `default` 后，再分别运行 A、B 的 `-once` 命令，B 会下载 A 上传的文件。

检查 B：

```sh
cat /tmp/xcloud-b/hello.txt
```

反向修改：

```sh
echo "hello from b" > /tmp/xcloud-b/hello.txt
```

B 上传：

```sh
XCLOUD_TOKEN=secret ./xcloud client \
  -root /tmp/xcloud-b \
  -server http://127.0.0.1:8080 \
  -space default \
  -device b \
  -once
```

A 下载：

```sh
XCLOUD_TOKEN=secret ./xcloud client \
  -root /tmp/xcloud-a \
  -server http://127.0.0.1:8080 \
  -space default \
  -device a \
  -once
```

检查 A：

```sh
cat /tmp/xcloud-a/hello.txt
```

## 部署建议

小规模内网测试可以直接运行服务端：

```sh
./xcloud server -addr :8080 -data /data/xcloud
```

然后通过 `/admin` 创建账号和 Space。客户端首次启动不需要 token，打开本地客户端页面登录云端账号后点击 `开启此账号同步` 即可。任意一端开启后，同账号下已登录的其他客户端会自动进入同步。客户端会把本地目录上报到后台，再由后台选择目录到 Space。

生产环境建议：

- 服务端放在内网或 VPN 后面。
- 外层使用 Nginx/Caddy/Traefik 终止 TLS。
- token 使用至少 32 字节随机值。
- 服务端数据目录定期备份。
- metadata.json 和 chunks 需要一起备份。
- 客户端同步目录避免包含系统目录、密钥目录、浏览器数据目录等高风险路径。

## 故障排查

### 401 unauthorized

客户端 token 不是任何启用账号的同步 token，或本地绑定的设备凭证已失效。检查：

```sh
XCLOUD_TOKEN
-token
```

无 token 启动模式下，可以打开本地客户端页面解除绑定后重新登录云端账号。旧 token 模式下，也可以在后台重置对应账号 token 后重新配置客户端。

### 文件没有同步

检查：

- 客户端是否正在运行。
- 如果使用单目录模式，是否使用了正确的 `-root`。
- 如果不传 `-root`，是否已在后台对上级目录点击 `展开下一级`，并等待客户端下一轮上报。
- 无 token 启动模式下，本地客户端页面是否已经登录云端账号；同账号任意客户端是否已经点击 `开启此账号同步`。
- 旧 token 模式下，是否使用了正确账号的同步 token。
- 客户端目录是否已经在后台选择到某个 Space。
- 多个目录是否被选择到了同一个 Space。
- 文件是否在 `.xcloud` 目录下。
- 文件是否是符号链接或特殊文件。
- 客户端日志是否有上传或下载错误。

### 本地删除没有传播

默认不会传播本地删除。需要启动客户端时加入：

```sh
-delete-remote
```

### 出现 conflict 文件

说明多个客户端对同一文件产生了并发修改。当前版本不会自动合并，需要人工检查两个文件内容后决定保留哪一份。

### 服务端 metadata.json 损坏

当前版本使用 JSON 文件做元数据存储。生产环境应定期备份，后续应迁移到 PostgreSQL。

## 开发说明

核心调用关系：

```text
cmd/xcloud/main.go
  ├── server.NewStore
  ├── server.New(...).Handler
  └── client.NewEngine(...).Run

client.Engine.SyncOnce
  ├── pullRemote
  ├── pushLocal
  └── pullRemote

server.Store.Commit
  ├── CleanRel
  ├── verifyManifest
  ├── conflict detection
  ├── append version
  └── append event
```

如果要扩展成生产版，优先把 `server.Store` 抽象成接口：

```go
type MetadataStore interface {
    Commit(req syncmodel.CommitRequest) (syncmodel.CommitResponse, error)
    Delete(req syncmodel.DeleteRequest) (syncmodel.CommitResponse, error)
    ListFiles() []syncmodel.FileEntry
    EventsAfter(after int64) []syncmodel.Event
}
```

再把 chunk 存储抽象成接口：

```go
type ChunkStore interface {
    Has(hash string) bool
    Put(hash string, data []byte) error
    Get(hash string) ([]byte, error)
}
```

这样可以逐步替换成 PostgreSQL + S3/MinIO，而不需要重写客户端协议。
