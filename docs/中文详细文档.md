# xcloud 中文详细文档

`xcloud` 是一个中心化文件同步 MVP。当前策略已经改为：**客户端只同步本机 xcloud 保存根目录下的内容**，不再让客户端上报任意本地目录，也不再由后台选择某个本地目录参与同步。

核心规则：

- 默认保存根目录是客户端进程启动目录下的 `xcloud`。
- 默认 Space 对应本地目录 `xcloud/default`。
- 其他 Space 对应本地目录 `xcloud/<space-id>`。
- 同账号、同 Space 的所有客户端必须保持相同文件内容。
- 文件新增、编辑、删除都会同步。
- 删除会进入云端回收站，保留 10 天。
- 管理后台可以从云端回收站恢复文件，恢复会同步到所有客户端。
- 管理后台仍然保留 Space，用来隔离不同项目。

## 架构

```text
Client A xcloud/default ─┐
Client B xcloud/default ─┼── xcloud Server ── metadata.json
Client C xcloud/default ─┘          │
                                    └── chunks/<sha256>
```

服务端负责：

- 账号登录和同步 token 鉴权。
- Space 隔离。
- 文件版本元数据。
- chunk 存储和 hash 校验。
- 同步事件分发。
- 删除 tombstone。
- 云端回收站和恢复。
- 同步记录。

客户端负责：

- 登录云端账号并保存本机凭证。
- 创建本机 xcloud 保存根目录。
- 按 Space 同步 `xcloud/<space-id>`。
- 扫描本地文件、计算 hash、上传缺失 chunk。
- 拉取远端事件、下载 chunk、安全写入本地。
- 实时监听文件变化，并用周期扫描兜底。
- 上报同步成功和失败记录。

## 当前能力

- 一个二进制同时支持服务端和客户端模式。
- 管理后台支持登录、注册、账号管理、Space 管理、客户端管理、同步设置、同步记录和云端回收站。
- 只有管理员账号可以管理其他账号。
- 普通账号只能管理自己的客户端、Space、同步设置和回收站。
- 每个账号拥有独立数据域。
- 每个账号可以创建多个 Space。
- 每个 Space 映射到客户端 xcloud 根目录下的一个子目录。
- 文件新增、修改、删除都在同账号同 Space 内同步。
- 删除文件在云端回收站保留 10 天。
- 回收站恢复会生成新的文件版本事件，并同步恢复到所有客户端。
- 服务端保存文件版本元数据和内容分块。
- 文件和分块均使用 SHA-256 校验。
- 上传前客户端先询问服务端缺失哪些块，只上传缺失块。
- 下载时客户端校验 chunk hash 和整文件 hash。
- 下载落盘使用临时文件 + 原子 rename，避免半写入文件覆盖原文件。
- 支持本地冲突保留，不静默覆盖本地未同步修改。
- 默认跳过客户端内部 `.xcloud` 状态目录、符号链接和特殊文件。

## 构建

在项目根目录执行：

```sh
go build -o xcloud ./cmd/xcloud
```

如果本机 Go 构建缓存目录受限，可以临时指定缓存目录：

```sh
GOCACHE=/tmp/go-build-cache go build -o xcloud ./cmd/xcloud
```

## 启动服务端

```sh
./xcloud server
```

参数：

```text
-config 服务端运行配置文件，默认 xcloud-server.json
-addr   可选；HTTP 监听地址，覆盖配置文件
-data   可选；服务端数据目录，覆盖配置文件
```

默认配置文件 `xcloud-server.json`：

```json
{
  "domain": "ixxmi.com",
  "port": 18002,
  "data_dir": "server-data"
}
```

管理后台的 `服务配置` 页面支持修改默认域名、端口和 data 目录。域名会立即用于页面提示；端口和 data 目录属于启动参数，需要重启服务端进程后生效。

服务端数据目录示例：

```text
server-data/
├── metadata.json
├── server.db
└── chunks/
    └── ab/
        └── cd/
            └── <sha256>
```

`server.db` 是 SQLite 数据库，保存服务配置、账号、客户端 token、客户端设备、Space 和同步设置等控制面信息。

`metadata.json` 继续保存文件树、版本、事件、回收站、同步记录和幂等操作结果；为了兼容历史数据，账号等控制面信息也会保留一份镜像。`chunks` 保存按内容 hash 命名的文件块。

## SQLite 数据库

服务端启动后会自动创建：

```text
<data目录>/server.db
```

当前 SQLite 表包括：

- `server_config`：管理端域名、端口、data 目录、监听主机。
- `accounts`：账号、密码 hash、管理员标记、禁用状态、同步开关、同步设置。
- `client_tokens`：客户端登录后签发的同步 token hash。
- `client_devices`：客户端设备、主机名、全局 xcloud 保存根目录、最后在线时间。
- `spaces`：账号下的 Space。

启动兼容策略：

1. 先读取 `metadata.json`，兼容旧版本数据。
2. 如果 `server.db` 已有账号等控制面数据，则以 SQLite 为准覆盖内存中的账号、客户端、Space 和同步设置。
3. 服务端后续修改控制面信息时，会同步写入 `metadata.json` 和 `server.db`。
4. 文件版本、事件、回收站、同步记录和 chunk 引用当前仍由 `metadata.json` 与 `chunks/` 保存。

因此备份时需要同时备份：

```text
server.db
metadata.json
chunks/
```

服务端启动后访问：

```text
http://ixxmi.com:18002/admin
```

首次启动会创建默认管理员：

```text
用户名：admin
密码：admin123
```

内网测试可以直接使用；正式部署必须修改初始化方式或立即修改默认密码，并放到 TLS 反向代理后面。

## 管理后台

后台主要菜单：

- `概览`：查看账号、Space、文件和回收站统计。
- `Space 与设备`：创建 Space，启停 Space，配置客户端全局 xcloud 保存根目录。
- `云端回收站`：查看 10 天内删除的文件并恢复。
- `同步记录`：查看上传、下载、删除、冲突、跳过、监听失败、扫描失败等记录。
- `账号管理`：仅管理员可见，用于创建、禁用、启用其他账号和重置 token。
- `同步设置`：配置实时监听、变更防抖时间、兜底扫描间隔。
- `服务配置`：仅管理员可见，用于配置管理端域名、端口和 data 目录。
- `安全设置`：修改当前登录密码，重置当前账号同步 token。

客户端保存根目录是设备级配置。例如后台把某台客户端的保存根目录改为：

```text
/data/xcloud
```

则该客户端会同步：

```text
/data/xcloud/default
/data/xcloud/<space-id>
```

后台允许配置客户端机器上的任意绝对路径。这是高权限能力，只应开放给可信账号。

## 账号和 Space 隔离

同步隔离规则：

```text
账号 alice + space default  <=>  账号 alice + space default
账号 alice + space docs     <=>  账号 alice + space docs
账号 bob   + space default  <=>  账号 bob   + space default
```

不同账号不会同步。即使两个账号都有 `default`，它们仍然是两个独立数据域。

同一账号下不同 Space 也不会互相同步：

```text
xcloud/default
xcloud/docs
xcloud/photos
```

这三个目录对应三个独立空间。

## 启动客户端：本机网页登录方式

首次启动客户端不需要 token：

```sh
mkdir -p ./xcloud
./xcloud client
```

客户端会启动本机管理页面：

```text
http://127.0.0.1:18080
```

操作流程：

1. 打开本机客户端页面。
2. 输入云端地址、账号或邮箱、密码。
3. 登录后点击 `开启此账号同步`。
4. 把要同步的文件放到 `xcloud/default`。
5. 同账号其他客户端也登录后，会自动开始同步。

本机客户端凭证保存到：

```text
~/.xcloud/client-config.json
```

开启同步是账号级云端开关。任意一台已经登录该账号的客户端点击开启后，其他已经登录同账号的客户端会在状态轮询中识别到同步已开启，并开始同步。

## 启动客户端：指定 xcloud 保存根目录

`-root` 现在表示客户端全局 xcloud 保存根目录，不表示要同步的任意业务目录。

```sh
mkdir -p /tmp/xcloud-a
./xcloud client -root /tmp/xcloud-a
```

默认 Space 文件放在：

```text
/tmp/xcloud-a/default
```

其他 Space 文件放在：

```text
/tmp/xcloud-a/<space-id>
```

如果不传 `-root`，默认使用：

```text
<客户端进程启动目录>/xcloud
```

## token 启动方式

旧的 token 启动方式仍然保留，适合脚本、后台服务或没有浏览器的环境。

先在后台重置或查看创建账号时显示的一次性同步 token，然后启动两台客户端：

```sh
mkdir -p /tmp/xcloud-a/default /tmp/xcloud-b/default
```

客户端 A：

```sh
./xcloud client \
  -root /tmp/xcloud-a \
  -server http://ixxmi.com:18002 \
  -token <account-token> \
  -device laptop-a
```

客户端 B：

```sh
./xcloud client \
  -root /tmp/xcloud-b \
  -server http://ixxmi.com:18002 \
  -token <same-account-token> \
  -device laptop-b
```

此时 `/tmp/xcloud-a/default` 和 `/tmp/xcloud-b/default` 会互相同步。

如果后台创建了 Space `docs`，客户端会同步：

```text
/tmp/xcloud-a/docs
/tmp/xcloud-b/docs
```

## 常用参数

```text
-root           可选；客户端全局 xcloud 保存根目录，默认 <工作目录>/xcloud
-server         服务端地址，默认 http://ixxmi.com:18002
-token          可选；账号同步 token，也可以通过 XCLOUD_TOKEN 设置；不传则启动本机客户端页面
-client-addr    本机客户端页面地址，默认 127.0.0.1:18080
-client-config  本机客户端配置路径，默认 ~/.xcloud/client-config.json
-space          兼容默认 Space ID，默认 default；实际启用 Space 会从服务端状态加载
-device         当前设备 ID，也可以通过 XCLOUD_DEVICE_ID 设置；默认由主机名和保存根目录生成
-state          Supervisor 状态文件路径，默认 ~/.xcloud/discovery-state.json
-interval       兜底扫描间隔，默认 10s；后台同步设置优先
-chunk-size     分块大小，默认 4194304，即 4 MiB
-once           只执行一轮同步后退出
-delete-remote  兼容旧参数；当前版本本地删除总是会同步到服务端
```

## 自动重启脚本

项目内置守护脚本：

```sh
./scripts/xcloud-watchdog.sh server
./scripts/xcloud-watchdog.sh client
```

脚本会自动构建两个独立进程名的二进制：

```text
bin/xclouds  服务端进程
bin/xcloudc  客户端进程
```

日志和运行锁：

```text
logs/xclouds.log
logs/xcloudc.log
run/xclouds.lock
run/xcloudc.lock
```

子进程异常退出后，脚本默认 3 秒后自动重启。可以通过环境变量调整：

```sh
XCLOUD_RESTART_DELAY=5 ./scripts/xcloud-watchdog.sh server
```

额外参数会透传给实际进程。例如指定客户端保存根目录：

```sh
./scripts/xcloud-watchdog.sh client -root /data/xcloud -server http://ixxmi.com:18002
```

服务端读取 `xcloud-server.json`，所以后续启动服务端不需要再指定端口或 data：

```sh
./scripts/xcloud-watchdog.sh server
```

## 同步范围

客户端只扫描各 Space 子目录：

```text
<xcloud保存根目录>/<space-id>/
```

示例：

```text
xcloud/default/report.docx
xcloud/default/images/a.png
xcloud/docs/readme.md
```

会同步。

下面内容不会作为业务文件同步：

```text
xcloud/default/.xcloud/
```

`.xcloud` 是客户端内部状态目录。

符号链接、设备文件、socket、pipe 等特殊文件也会跳过。普通隐藏文件不会因为文件名以 `.` 开头而跳过，只有 `.xcloud` 状态目录被保留为内部目录。

## 实时同步和兜底扫描

客户端启动后会：

1. 向服务端查询账号同步开关、同步设置、客户端保存根目录和启用 Space。
2. 为每个启用 Space 创建本地目录。
3. 拉取远端事件。
4. 扫描本地文件并上传新增、修改、删除。
5. 再拉取一次远端事件，处理同步过程中出现的新远端变化。
6. 如果启用实时监听，则监听文件变化并防抖触发同步。
7. 即使实时监听失败，也会按兜底扫描间隔继续同步并上报失败记录。
8. 如果网络短暂断开，单次 HTTP 请求会自动重试；如果重试耗尽，本轮同步失败但客户端进程继续运行，下一次实时触发或兜底扫描会再次连接服务端。

后台可配置：

```text
实时监听：开启 / 关闭
变更防抖毫秒：默认 800
兜底扫描间隔秒：默认 10
```

## 自动重连

客户端当前是 HTTP 轮询 + 文件系统监听模型，不依赖常驻长连接。每个同步请求都会对临时故障做自动重试：

- 网络错误。
- 请求超时。
- HTTP `429 Too Many Requests`。
- HTTP `5xx` 服务端错误。

HTTP `400`、`401`、`403`、`404` 这类确定性客户端错误不会重试，会直接返回失败原因，避免密码、权限或路径错误被无意义地重复请求。

重试使用指数退避。短暂断网、网关重启、服务端热重启时，客户端通常会在同一轮同步里恢复。如果所有重试都失败：

- 当前同步轮次失败并记录日志。
- 客户端不会退出。
- 实时监听继续保持；文件变化会继续触发新的同步轮次。
- 兜底扫描继续按后台配置的间隔运行。
- 网络恢复后，客户端从本地保存的最后事件序号继续拉取服务端事件，同时重新扫描 `xcloud/<space-id>`，补齐断网期间的新增、编辑和删除。

因此断链恢复后不需要人工重启客户端。只要账号同步开关仍然开启，客户端会在下一次成功连接后继续同步。

## 删除和云端回收站

本地删除现在总是会同步。

删除流程：

```text
用户删除 xcloud/default/a.txt
客户端扫描发现本地文件消失
客户端向服务端提交 delete
服务端生成 deleted 版本事件
其他客户端拉取事件后删除本地文件
服务端回收站保留该文件 10 天
```

回收站恢复流程：

```text
管理员或账号本人在 /admin 的云端回收站点击恢复
服务端找到删除前最后一个 file 版本
服务端生成新的 file 版本事件
所有客户端拉取事件并重新下载该文件
```

恢复不是把旧事件撤销，而是生成一个新的文件版本。这样客户端只需要按正常事件流处理即可。

## 同步记录

客户端会上报文件操作级同步记录，包括：

- 上传成功或失败。
- 下载成功或失败。
- 删除成功或失败。
- 冲突保留。
- 跳过已处理版本。
- 文件监听失败。
- 扫描失败。

每条记录包含：

```text
账号
Space
设备
根目录
文件路径
动作
状态
错误信息
耗时
时间戳
```

服务端默认每账号保留最近 5000 条记录，避免 `metadata.json` 无限增长。

## 同步流程

单轮同步顺序：

```text
1. pullRemote   拉取服务端事件并应用到本地
2. pushLocal    扫描 xcloud/<space> 并上传本地新增、修改、删除
3. pullRemote   再拉一次，处理上传期间出现的新远端变化
```

### 本地扫描

客户端遍历当前 Space 目录：

```text
<xcloud保存根目录>/<space-id>
```

扫描规则：

- 跳过 `.xcloud` 状态目录。
- 跳过符号链接。
- 跳过目录、设备文件、socket、pipe 等特殊文件。
- 只处理普通文件。
- 对每个普通文件计算整文件 SHA-256。
- 按 `-chunk-size` 分块并计算每个 chunk 的 SHA-256。

### 上传流程

```text
客户端扫描文件
客户端计算 manifest
客户端调用 /v1/chunks/check
服务端返回缺失的 chunk hash
客户端上传缺失 chunk
客户端调用 /v1/commit 提交版本
服务端校验 manifest
服务端生成新版本和事件
```

manifest 概念：

```json
{
  "path": "docs/a.txt",
  "base_version": "旧版本 ID",
  "size": 123,
  "hash": "整文件 sha256",
  "chunks": [
    {
      "index": 0,
      "hash": "chunk sha256",
      "size": 123
    }
  ],
  "mod_time_unix": 1710000000
}
```

服务端提交版本时会重新读取对应 chunk，重新计算整文件 size 和 hash。只有服务端计算结果与 manifest 一致时，版本才会提交成功。

### 下载流程

```text
客户端拉取服务端事件
发现远端有新 file 版本
检查本地是否存在未同步修改
如有未同步修改，先改名保留为冲突副本
下载所有 chunk 到临时文件
校验 chunk hash
校验整文件 hash
原子 rename 覆盖目标文件
更新本地 state
```

## 版本和冲突

每个文件有稳定的 `file_id`，每次修改生成新的 `version_id`。

客户端提交修改时会带上本地已知的 `base_version`：

```text
base_version == 服务端当前版本
=> 正常提交

base_version != 服务端当前版本
=> 服务端生成冲突副本
```

服务端提交冲突示例：

```text
report.docx
report (conflict from laptop-b at 20260516-103000).docx
```

客户端拉取冲突示例：

```text
report.docx
report (local conflict from laptop-a at 20260516-103000).docx
```

当前版本不做文本三方合并，选择保守保留两份文件。

## 客户端状态文件

每个 Space 的状态文件在对应 Space 目录下：

```text
<xcloud保存根目录>/<space-id>/.xcloud/state-<space-id>.json
```

它记录：

- 本地路径。
- file_id。
- version_id。
- 文件状态，`file` 或 `deleted`。
- 文件大小。
- 文件 hash。
- chunk 列表。
- 上次处理到的服务端事件序号。

Supervisor 状态文件默认在：

```text
~/.xcloud/discovery-state.json
```

这个文件名保留了早期目录发现模式的命名，但当前版本不再用它保存目录上报状态。

不要手动编辑这些状态文件，除非你明确知道会造成什么结果。

## 服务端 HTTP API

主要 API：

```text
GET  /healthz
POST /v1/client/login
GET  /v1/client/status
POST /v1/client/sync
POST /v1/sync-records
GET  /v1/files
GET  /v1/events?after=<seq>
POST /v1/chunks/check
PUT  /v1/chunks/<hash>
GET  /v1/chunks/<hash>
POST /v1/commit
POST /v1/delete
```

兼容旧客户端的目录上报 API 仍保留：

```text
POST /v1/folders/report
GET  /v1/folders/status
POST /v1/folders/children-complete
```

新客户端不再调用这些目录上报接口。

除 `/healthz` 和 `/v1/client/login` 外，同步接口需要：

```text
Authorization: Bearer <token>
X-XCloud-Space: <space_id>
X-XCloud-Device: <device_id>
X-XCloud-Root: <absolute_space_root_path>
```

`X-XCloud-Space` 为空时默认使用 `default`。

### POST /v1/client/login

客户端本机页面使用云端账号密码换取本机同步 token。

请求概念：

```json
{
  "identifier": "alice",
  "password": "password",
  "device_id": "laptop-a",
  "hostname": "my-host",
  "storage_root": "/tmp/xcloud-a"
}
```

返回账号信息、设备 token、同步开关、同步设置和服务端配置的保存根目录。

### GET /v1/client/status

客户端轮询账号同步状态。

返回：

- 当前账号。
- 同步是否开启。
- 同步触发设置。
- 客户端设备保存根目录。
- 启用中的 Space 列表。

客户端据此启动或停止每个 Space 的同步。

### POST /v1/client/sync

开启或关闭账号级同步开关。

任意一个已登录客户端开启后，同账号其他客户端会自动感知。

### POST /v1/sync-records

客户端上报文件操作级同步记录。

### GET /v1/files

返回服务端当前文件列表，包括已删除条目的 tombstone 状态。

### GET /v1/events

返回某个事件序号之后的所有事件。客户端通过它增量拉取远端变化。

### POST /v1/chunks/check

客户端提交一组 chunk hash，服务端返回缺失的 hash。客户端只上传缺失块。

### PUT /v1/chunks/<hash>

上传一个 chunk。服务端会计算上传内容的 SHA-256，必须与 URL 中的 hash 一致。

### GET /v1/chunks/<hash>

下载一个 chunk。客户端下载后还会再次校验 hash。

### POST /v1/commit

提交一个文件版本。服务端会验证：

- operation_id 是否存在。
- device_id 是否存在。
- 路径是否是安全相对路径。
- 文件 hash 是否合法。
- 每个 chunk 是否存在。
- chunk index 是否连续。
- chunk size 是否匹配。
- 由 chunk 重新计算出的整文件 hash 是否匹配。
- base_version 是否与当前版本一致。

### POST /v1/delete

提交一个删除 tombstone。服务端会验证 base_version，避免删除掉已经被其他客户端更新过的文件。

删除版本会保留被删除文件的 size、hash 和 chunks，供 10 天内恢复使用。

## 安全设计

当前 MVP 已实现：

- Bearer token 鉴权。
- 同步 token 绑定账号。
- 文件、事件、版本和操作幂等结果按账号和 Space 隔离。
- 非登录接口按账号校验数据域。
- chunk 下载需要账号拥有该 chunk 的授权。
- 服务端路径校验，拒绝绝对路径、空路径、`.`、`..` 和路径穿越。
- 服务端校验 chunk hash。
- 服务端重新计算整文件 hash。
- 客户端下载后校验 chunk 和整文件 hash。
- 客户端跳过符号链接，避免把同步目录外的文件意外带入。
- 客户端写文件使用临时文件和原子 rename。

生产环境仍然必须补齐：

- TLS 1.3，避免 token 和文件内容明文传输。
- 更好的设备注册和吊销机制。
- 每设备独立凭证或 mTLS 客户端证书。
- 管理后台 CSRF 防护。
- 服务端 API 限流和审计日志。
- Space 级更细粒度权限控制。
- 端到端加密，服务端只保存密文。
- 文件名和路径元数据加密，适用于高安全场景。

## 可靠性设计

当前版本具备：

- 客户端 state 记录已同步版本。
- 服务端 `operation_id` 支持幂等提交，避免客户端重试造成重复版本。
- 文件内容先上传 chunk，再提交 manifest。
- 只有 commit 成功后，其他客户端才会看到新事件。
- 下载文件先写临时文件，校验成功后原子替换。
- 删除作为版本事件保存，并进入 10 天回收站。
- 恢复通过新增版本事件同步到所有客户端。
- 冲突时保留副本，不静默覆盖。
- fsnotify 实时监听 + 周期扫描兜底。
- 客户端对网络错误、超时、HTTP `429` 和服务端 `5xx` 自动重试。
- 重试耗尽后客户端仍保持运行，网络恢复后继续从事件序号和本地扫描补齐同步。

后续可继续增强：

- 客户端本地任务队列，明确记录 pending upload/download/delete。
- 服务端元数据迁移到 PostgreSQL，用事务保证提交一致性。
- chunk 存储迁移到 MinIO/S3，并增加生命周期管理。
- 增加历史版本保留策略。
- 增加快照和目录级回滚。
- 增加断点续传和并发上传下载。
- 同步空目录的独立目录元数据。

## 当前限制

这版代码是 MVP，不是最终生产版。

已知限制：

- 当前以普通文件同步为核心；空目录没有独立版本元数据。
- 客户端已经接入 fsnotify 实时监听，并使用周期性扫描作为兜底；离线期间仍依赖客户端恢复运行后的扫描补齐。
- 服务端元数据使用单个 JSON 文件，不适合大规模并发和海量文件。
- Bearer token 仍是共享密钥，设备级吊销能力还不完整。
- 管理后台 session 保存在服务端内存中，服务重启后需要重新登录。
- 当前没有找回密码和邮箱验证能力。
- 没有内置 TLS，需要通过反向代理或后续代码补齐。
- 没有端到端加密。
- 旧 chunk 暂不会自动清理。
- 没有文本三方合并。
- 没有带宽限速、并发控制和系统服务安装脚本。

## 本地验证流程

启动服务端：

```sh
./xcloud server
```

打开 `/admin` 注册或登录账号，重置同步 token，下面命令里的 `<token>` 替换为该 token。

创建两个客户端根目录：

```sh
mkdir -p /tmp/xcloud-a/default /tmp/xcloud-b/default
```

在 A 写入文件：

```sh
echo "hello from a" > /tmp/xcloud-a/default/hello.txt
```

A 上传：

```sh
./xcloud client \
  -root /tmp/xcloud-a \
  -server http://ixxmi.com:18002 \
  -token <token> \
  -device a \
  -once
```

B 下载：

```sh
./xcloud client \
  -root /tmp/xcloud-b \
  -server http://ixxmi.com:18002 \
  -token <token> \
  -device b \
  -once
```

检查 B：

```sh
cat /tmp/xcloud-b/default/hello.txt
```

反向修改：

```sh
echo "hello from b" > /tmp/xcloud-b/default/hello.txt
```

B 上传：

```sh
./xcloud client \
  -root /tmp/xcloud-b \
  -server http://ixxmi.com:18002 \
  -token <token> \
  -device b \
  -once
```

A 下载：

```sh
./xcloud client \
  -root /tmp/xcloud-a \
  -server http://ixxmi.com:18002 \
  -token <token> \
  -device a \
  -once
```

检查 A：

```sh
cat /tmp/xcloud-a/default/hello.txt
```

验证删除：

```sh
rm /tmp/xcloud-a/default/hello.txt

./xcloud client -root /tmp/xcloud-a -server http://ixxmi.com:18002 -token <token> -device a -once
./xcloud client -root /tmp/xcloud-b -server http://ixxmi.com:18002 -token <token> -device b -once

test ! -e /tmp/xcloud-b/default/hello.txt
```

然后在 `/admin` 的 `云端回收站` 中点击恢复，再分别运行 A、B 的 `-once`，两个客户端都会重新出现该文件。

## 部署建议

小规模内网测试可以直接运行服务端：

```sh
./xcloud server
```

然后通过 `/admin` 创建账号和 Space。客户端首次启动不需要 token，打开本地客户端页面登录云端账号后点击 `开启此账号同步` 即可。

生产环境建议：

- 服务端放在内网或 VPN 后面。
- 外层使用 Nginx/Caddy/Traefik 终止 TLS。
- token 使用至少 32 字节随机值。
- 服务端数据目录定期备份。
- metadata.json 和 chunks 需要一起备份。
- 客户端 xcloud 保存根目录避免放系统目录、密钥目录、浏览器数据目录等高风险路径。
- 管理后台允许配置客户端机器上的任意绝对路径作为设备保存根目录；只应开放给可信账号。

## 故障排查

### 401 unauthorized

客户端 token 不是有效同步 token，或本地绑定的设备凭证已失效。检查：

```sh
XCLOUD_TOKEN
-token
```

无 token 启动模式下，可以打开本地客户端页面解除绑定后重新登录云端账号。

### 文件没有同步

检查：

- 客户端是否正在运行。
- 本地客户端页面是否已经登录云端账号。
- 同账号任意客户端是否已经点击 `开启此账号同步`。
- 旧 token 模式下是否使用了正确账号的同步 token。
- 文件是否放在 `<xcloud保存根目录>/<space-id>` 下，例如 `xcloud/default`。
- 目标 Space 是否在后台启用。
- 两台客户端是否登录同一个账号。
- 文件是否位于 `.xcloud` 内部状态目录。
- 文件是否是符号链接或特殊文件。
- 客户端日志是否有上传、下载、监听或扫描错误。
- 管理后台 `同步记录` 是否显示失败原因。

### 删除后想恢复

打开 `/admin`，进入 `云端回收站`，在 10 天内点击恢复。恢复后客户端下一轮同步会重新下载该文件。

### 出现 conflict 文件

说明多个客户端对同一文件产生了并发修改。当前版本不会自动合并，需要人工检查两个文件内容后决定保留哪一份。

### 服务端 metadata.json 损坏

当前版本使用 JSON 文件做元数据存储。生产环境应定期备份，后续应迁移到 PostgreSQL。

## 开发说明

核心调用关系：

```text
cmd/xcloud/main.go
  ├── server.NewStore
  ├── server.New(...).Handler
  └── client.NewSupervisor(...).Run

client.Supervisor.SyncOnce
  ├── /v1/client/status
  ├── 为每个启用 Space 创建 Engine
  └── 启动或执行 Engine.SyncOnce

client.Engine.SyncOnce
  ├── pullRemote
  ├── pushLocal
  └── pullRemote

server.Store.Commit
  ├── CleanRel
  ├── verifyManifest
  ├── conflict detection
  ├── append version
  └── append event

server.Store.Delete
  ├── validate base version
  ├── append deleted version
  └── keep restorable chunks for trash retention

server.Store.RestoreTrash
  ├── find latest file version before delete
  ├── append restored file version
  └── append event for all clients
```

后续生产化可先抽象元数据存储：

```go
type MetadataStore interface {
    Commit(accountID, spaceID string, req syncmodel.CommitRequest) (syncmodel.CommitResponse, error)
    Delete(accountID, spaceID string, req syncmodel.DeleteRequest) (syncmodel.CommitResponse, error)
    RestoreTrash(accountID string, req syncmodel.RestoreRequest, actor string) (syncmodel.CommitResponse, error)
    ListFiles(accountID, spaceID string) []syncmodel.FileEntry
    EventsAfter(accountID, spaceID string, after int64) []syncmodel.Event
}
```

再抽象 chunk 存储：

```go
type ChunkStore interface {
    Has(hash string) bool
    Put(hash string, data []byte) error
    Get(hash string) ([]byte, error)
}
```

这样可以逐步替换成 PostgreSQL + S3/MinIO，而不需要重写客户端协议。
